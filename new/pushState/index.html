Ghostbusters

Inspired from https://css-tricks.com/using-the-html5-history-api/, this is a test of vanilla javascript routing with pushState without any server-side backing. Can testindex.html serve as the main SPA page with supporting content pages allowing browser refreshes?
With pushState, we can push fake URLs into history. However doing so creates a problem whereby page refreshes of those fake URLs result in 404s. I'm trying to work around the 404s by actually creating content at the locations e.g. new/pushState/egon/index.html for new/pushState/egon. We need to handle scenarios:
  1. User visits main page first, and clicks to navigate to content pages
  2. User visits content page first, or refreshes the browser
Scenario 1 is handled by pushState logic in the main page i.e. new/pushState/index.html.
Scenario 2 is handled by some simple javascript-based redirect code plus some code that auto-triggers Scenario 1.
